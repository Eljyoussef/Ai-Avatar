DETAILED SERVICE ARCHITECTURE
1. NATS — Event Mesh (Infrastructure)
Technology: NATS Server 2.10 with JetStream

Purpose: Central nervous system — every service communicates exclusively through NATS. No direct HTTP between services.

Subject Hierarchy:

text
session.{id}.audio.input          # Raw audio from user (binary)
session.{id}.audio.output         # TTS audio to user (binary)
session.{id}.video.output         # Avatar video frames (binary)
session.{id}.asr.partial          # Partial transcription
session.{id}.asr.final            # Complete transcription  
session.{id}.intent.detected      # Classified intent
session.{id}.rag.query            # Retrieval request
session.{id}.rag.results          # Retrieved documents + scores
session.{id}.llm.token            # Streaming LLM tokens
session.{id}.tts.chunk            # Synthesized audio chunks
session.{id}.avatar.render        # Avatar render request
session.{id}.control.interrupt    # Barge-in signal
session.{id}.accent.detected      # Detected accent
session.{id}.accent.update        # Accent profile update
session.{id}.memory.update        # Save to persistent memory
session.{id}.output               # Final output to client
Why NATS over Kafka/RabbitMQ:

Sub-millisecond latency (critical for voice)

10M messages/second throughput

Built-in JetStream for persistence

Request-reply pattern for RAG queries

No Zookeeper/JVM overhead

2. Gateway (FastAPI + WebSocket)
Technology: Python 3.11, FastAPI, WebSocket, WebRTC

Flow:

text
Browser (WebRTC) → Gateway WebSocket → NATS
                                    ↕
Browser ← Gateway WebSocket ← NATS
Responsibilities:

WebRTC Signaling: SDP exchange for peer-to-peer audio

WebSocket Management: Real-time bidirectional channel per session

JWT Authentication: Token validation on connection

Audio Routing: Forward raw audio bytes to NATS audio.input

Response Relay: Subscribe to NATS session.{id}.output → forward to WebSocket

Rate Limiting: Per-session message throttling

Health Checks: /health endpoint for monitoring

Key Implementation Detail:
The gateway is stateless — all session state lives in Redis via Session Manager. This allows horizontal scaling.

3. Session Manager (Go)
Technology: Go 1.22, Redis, NATS

Why Go: Sub-millisecond state mutations needed for barge-in handling. Python's GIL would bottleneck concurrent sessions.

State per Session:

go
type SessionState struct {
    ID              string            // UUID
    UserID          string            // Authenticated user
    Gender          string            // male/female/neutral
    AccentProfile   string            // fr-FR/fr-CA/fr-AF/fr-BE/fr-CH/fr-MG
    IsActive        bool              // Session alive?
    IsInterrupted   bool              // Barge-in flag
    IsSpeaking      bool              // TTS currently playing?
    AudioBuffer     []byte            // Last 10s of raw audio
    MessageHistory  []Message         // Last 50 messages
    ActiveDocuments []string          // Current document context
    Preferences     UserPreferences   // Voice speed, avatar mode
    TTL             time.Duration     // 30-minute expiry
}
Core Operations:

CreateSession(userID) → Returns session UUID

HandleInterrupt(sessionID) → Sets interrupt flag, publishes to NATS

SetAccent(sessionID, accent) → Updates accent profile

AddMessage(sessionID, msg) → Appends to history (max 50)

CleanupExpired() → Background goroutine every 5 minutes

Barge-in Flow:

text
User clicks "stop" → Gateway publishes interrupt signal
→ Session Manager sets IsInterrupted=true
→ Publishes to session.{id}.control.interrupt
→ Inference Gateway stops token generation
→ TTS Service stops audio synthesis
→ Audio buffer cleared
4. Accent Router (Python + XLS-R)
Technology: Wav2Vec2-XLS-R 300M fine-tuned on 6 French variants

Model Details:

Base: facebook/wav2vec2-xls-r-300m

Fine-tuned on: Common Voice French + PFC + CIEL-F + African Accents Corpus

Output: 6-class classification (fr-FR, fr-CA, fr-AF, fr-BE, fr-CH, fr-MG)

Latency: <50ms on CPU for 2-second audio chunk

Flow:

text

Early Intent Detection:

text

    ↓
Intent Service: {"intent": "question", "confidence": 0.87}
    ↓
Triggers RAG Prefetch BEFORE user finishes speaking
    ↓

    ↓
LLM receives query + documents simultaneously
This cuts perceived latency by 40-60%.

6. ASR Service (Faster-Whisper)
Technology: Faster-Whisper large-v3 with CTranslate2 quantization

Model Specs:

1.5B parameters

INT8 quantization (4x faster, 2x smaller)

VAD: WebRTC Voice Activity Detector (mode 3)

Endpointing: 500ms silence threshold

Streaming: Partial transcripts every 100ms

Streaming Flow:

text
Audio chunk (100ms) 
    ↓
VAD: Is speech? 
    ↓ YES
Whisper transcribe buffer → partial text
    ↓
Publish: session.{id}.asr.partial {"text": "je voudrais sav...", "is_partial": true}
    ↓
VAD: Silence >500ms?
    ↓ YES (endpoint)
Whisper transcribe full buffer → final text
    ↓
Publish: session.{id}.asr.final {"text": "je voudrais savoir comment fonctionne l'assurance", "is_partial": false}
    ↓
Reset audio buffer
Accent-Specific Models:
Each accent loads a different Whisper fine-tune with vocabulary bias injection for regional terms.

7. TTS Service (XTTS-v2)
Technology: Coqui TTS XTTS-v2 with multi-accent voice embeddings

Model Specs:

1.5B parameters

24kHz sample rate

Streaming: phrase-level synthesis (not word-level)

Voice cloning: 6-second reference audio

Streaming Synthesis Flow:

text
LLM Token Stream: ["L'", "assurance", "maladie", "couvre", "les", "soins", "."]
    ↓
Token Buffer accumulates until phrase boundary (./!/?/:)
    ↓
Phrase: "L'assurance maladie couvre les soins."
    ↓
XTTS-v2 synthesize with accent-specific voice embedding
    ↓
Publish: session.{id}.audio.output → WAV bytes
    ↓
Publish: session.{id}.avatar.render → WAV bytes (if avatar enabled)
    ↓
Clear buffer, continue with next tokens
Voice Mapping:

text
Accent + Gender → Voice Embedding
fr-FR + female  → parisian_female_v2.wav
fr-CA + male    → quebec_male_v2.wav  
fr-AF + female  → abidjan_female_v2.wav
8. RAG Service (Hybrid Retrieval)
Technology: Qdrant (dense) + OpenSearch (sparse) + Cross-Encoder reranking + LLMLingua compression

Retrieval Pipeline:

text
Query: "Comment fonctionne l'assurance maladie?"
    ↓
┌─────────────────────────────────────────┐
│ PARALLEL RETRIEVAL                      │
│                                         │
│ Dense (Qdrant)          Sparse (OS)     │
│ multilingual-e5-large   BM25 + SPLADE   │
│ Top-20 @ 0.92          Top-20 @ 0.87   │
└─────────────────────────────────────────┘
    ↓
Reciprocal Rank Fusion (RRF)
    ↓ Top-10 candidates
Cross-Encoder Reranker (mmarco-mMiniLMv2)
    ↓ Top-5 documents
LLMLingua Contextual Compression
    ↓ Compressed to fit context window
Return: 3 compressed document chunks
Chunking Strategy (Document Processor):

Level 1: Structural — split by headings/sections

Level 2: Semantic — 300-600 token chunks with 50-token overlap

Level 3: Contextual — LLMLingua compression before prompt insertion

10. Memory Service (Multi-Tier)
3-Tier Memory Architecture:

Tier	Storage	Purpose	TTL
Hot	Redis	Current conversation, session state	30 min
Warm	PostgreSQL	Full transcripts, user preferences	Forever
Cold	Qdrant	Semantic facts, user profile embeddings	Forever
Hot Memory (Redis):

text
Key: session:{id}:messages → List of last 20 messages (JSON)
Key: session:{id}:prefs → User preferences (JSON)
Key: session:{id}:interrupt → Interrupt flag (boolean)
Warm Memory (PostgreSQL Schema):

sql
users: id, email, gender, preferred_accent, created_at
conversations: id, user_id, session_id, started_at, ended_at
messages: id, conversation_id, role, content, documents_used, latency_ms
documents: id, user_id, filename, chunk_count, processing_status
user_preferences: id, user_id, voice_speed, response_style, avatar_quality
Cold Memory (Qdrant):

Embeddings of user facts extracted from conversations

Every 10 messages → summarizer extracts key facts → embed → store in Qdrant

Query: search_facts(user_id, "préférences assurance") → relevant user context

11. Avatar Adapter (Duix Integration)
Technology: Duix rendering engine, WebRTC video streaming, Lip-sync analysis

Two Rendering Modes:

Mode 1 — Realtime (Lightweight):

text
Audio chunk → Analyze amplitude
    ↓
Generate viseme data (mouth openness, phoneme class)
    ↓
Send to browser → Canvas-based 2D animation
    ↓
Latency: <50ms
Mode 2 — HD Avatar (Duix):

text
Audio chunk → Duix Renderer
    ↓
Generate photorealistic talking head frame
    ↓
Stream via WebRTC video channel
    ↓
Latency: 500ms-1s
Lip-Sync Analysis:

text
Audio frame (16ms) → FFT → Energy + Zero-crossing rate
    ↓
Map to mouth openness (0.0 to 1.0)
    ↓
Classify phoneme: silence / consonant / vowel
    ↓
Publish viseme sequence to client
COMPLETE REQUEST FLOW (End-to-End)
text
1. USER SPEAKS: "Comment fonctionne l'assurance maladie?"
   ↓ WebRTC → Gateway → NATS session.abc.audio.input

2. SESSION MANAGER: Checks session state, accent profile, gender
   ↓ Routes to Accent Router

3. ACCENT ROUTER: Classifies audio → "fr-CA" (Québec)
   ↓ Updates session accent, selects ASR/TTS profiles

4. ASR SERVICE: 
   - Partial: "comment fonctionne..." → Intent Service
   - Final: "comment fonctionne l'assurance maladie"

5. INTENT SERVICE (triggered by partial):
   - Detects: "question" intent
   - Triggers RAG prefetch immediately

6. RAG SERVICE (parallel to ASR completion):
   - Dense search (Qdrant) + Sparse search (OpenSearch)
   - RRF fusion → Cross-encoder rerank → Compress
   - Returns top 3 document chunks

7. INFERENCE GATEWAY:
   - Builds gendered prompt: "Tu parles à un homme québécois..."
   - Inserts documents + history
   - vLLM streams tokens

8. TTS SERVICE:
   - Receives token stream
   - Synthesizes phrase-by-phrase with Quebec male voice
   - Publishes audio chunks

9. AVATAR ADAPTER:
   - Analyzes audio for lip-sync visemes
   - Renders talking avatar (realtime or HD)

10. GATEWAY → CLIENT:
    - Audio chunks via WebSocket
    - Avatar frames via WebRTC video
    - Text response displayed in chat

11. MEMORY SERVICE:
    - Saves to Redis (hot)
    - Persists to PostgreSQL (warm)
    - Extracts semantic facts every 10 turns → Qdrant (cold)
